# Test-Driven Development：TDD + L4 验证

## 技能类型：刚性

严格遵循 RED-GREEN-REFACTOR 循环。不能跳过任何阶段。**跳过 TDD = 用用户的时间做你的调试器。**

---

## 哲学：测试是 brainstorming 的延续，不是单独的阶段

**测试场景设计发生在 brainstorming，TDD 负责执行。** TDD 不重新设计测试场景——它从 brainstorming 产出中加载，然后转化为可执行的测试用例。

```
brainstorming（设计测试场景）          TDD（执行测试）
    │                                     │
    ├─ 多用户/隔离场景 ──────────→ 写隔离测试（用户A数据≠用户B）
    ├─ 边界条件场景   ──────────→ 写边界测试（null/空/极值）
    ├─ 并发/竞态场景  ──────────→ 写并发测试（重复提交/同时编辑）
    └─ 异常路径场景   ──────────→ 写异常测试（超时/服务挂了/非法参数）
```

**TDD 不负责想测试场景 — 那是 brainstorming 的工作。TDD 负责把场景变成可运行的测试代码。**

## 前置检查清单（写任何代码之前逐项打勾）

**以下 5 项全部打勾才能写第一行实现代码。这不是可选的 — 这是刚性技能的入口条件。**

```
┌─────────────────────────────────────────────────────────────┐
│         写代码之前，先逐项核对以下清单                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  □ 1. 已从 brainstorming 加载测试场景？                       │
│     → 打开 knowledge-base/{项目名}/{项目名}-roadmap.md         │
│     → 找到本次任务的设计大纲 → 「测试场景设计」栏               │
│     → 4 类场景（多用户隔离/边界/并发/异常）每类 ≥2 条？         │
│     → 如果 brainstorming 没产出测试场景 → 回到 brainstorming    │
│       补全后再进 TDD                                         │
│                                                             │
│  □ 2. L4 签名已验证？                                         │
│     → 本次要调用的关键类/方法，打开源码确认签名                  │
│     → 返回类型真的是你想要的吗？（Optional<T> vs T？）         │
│     → 参数顺序对吗？异常声明对吗？                              │
│     → 至少验证 3 个关键方法签名                                │
│                                                             │
│  □ 3. 测试用例清单已写出？                                    │
│     → 基于 brainstorming 的 4 类场景，每类转化为 ≥2 个测试方法  │
│     → 测试方法名表达意图（should_xxx_when_yyy）               │
│     → 覆盖：正常路径 + 异常路径 + 边界 + 隔离                  │
│                                                             │
│  □ 4. 测试已全部运行且至少一个失败（RED）？                     │
│     → 如果有测试直接通过 → 测试写错了，或行为已存在              │
│     → 如果全部失败 → RED 阶段完成，进入 GREEN                  │
│                                                             │
│  □ 5. 确认"故意失败"不是"测试写错"？                          │
│     → 失败原因是"被测代码还没写"而不是"测试逻辑错了"            │
│     → 检查：手动在实现类里 return 正确值 → 测试应该变绿          │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│  ⚠ 以上 5 项全部打勾才能写第一行实现代码                        │
│  ⚠ 缺少任何一项 → 先补完再往下走                                │
└─────────────────────────────────────────────────────────────┘
```

## RED-GREEN-REFACTOR 循环

```
┌──────────────────────────────────────────────────────────┐
│  RED：基于 brainstorming 场景写失败的测试                    │
│                                                          │
│  输入：brainstorming 产出的 4 类测试场景                    │
│  输出：≥8 个测试方法（每类 ≥2 个）全部失败（红）            │
│                                                          │
│    多用户/隔离 → test_userA_cannot_see_userB_data()       │
│    边界条件   → test_should_reject_null_userId()          │
│    并发/竞态  → test_duplicate_submit_is_idempotent()     │
│    异常路径   → test_should_handle_db_timeout_gracefully()│
│                                                          │
│    → 运行测试 → 确认全部失败（红）                         │
│    → 如果某个测试直接通过 → 测试写错了（不是被测行为已实现）  │
│    → 提交：git commit -m "RED: {场景描述}"                 │
│                                                          │
├──────────────────────────────────────────────────────────┤
│  GREEN：写最小代码让测试通过                                │
│                                                          │
│    → 只写让当前这批测试通过的代码                           │
│    → 不做任何额外的事情（YAGNI：You Ain't Gonna Need It）   │
│    → 不要趁机"顺便"加个字段、重构个方法                     │
│    → 运行测试 → 确认全部通过（绿）                         │
│    → ★ L4 签名验证：调用的关键类方法签名与源码一致吗？       │
│       至少用 Read 工具验证 3 个方法的实际签名               │
│    → 提交：git commit -m "GREEN: {场景描述}"               │
│                                                          │
├──────────────────────────────────────────────────────────┤
│  REFACTOR：清理代码                                       │
│                                                          │
│    → 消除重复（DRY）                                      │
│    → 改善命名（表达意图）                                  │
│    → 提取常量/方法（可读性）                               │
│    → 保持测试绿色（每改一步运行一次测试）                    │
│    → 提交：git commit -m "REFACTOR: {清理描述}"            │
└──────────────────────────────────────────────────────────┘
```

## L4 签名验证（GREEN 阶段强制执行）

在 GREEN 阶段完成后、声称"通过了"之前，验证你使用的关键类：

```
你调用了 UserService.getById(Long id)？
  → 打开 UserService.java → 确认方法签名真的是 getById(Long id)
  → 确认返回类型是 User 还是 Optional<User>
  → 确认异常声明（throws XxxException？）
  → 如果 L3 记录不一致 → 以 L4 为准，标注 L3 待更新
```

**为什么要这样做**：LLM 凭记忆写的方法签名有约 15% 的概率是错的（参数名、返回类型、异常声明）。L4 验证将这 15% 的错误在 REFACTOR 之前拦截。

**至少验证 3 个本次调用的关键方法**。不是"我记得"——打开源码看。

## Bug 修复的 TDD 模式

修复 bug 时，TDD 的作用是：**先证明 bug 存在 → 再证明 bug 已修复 → 最后确保没有引入新 bug。**

```
bug 报告 / 发现问题
    │
    ▼
RED：写一个复现 bug 的测试
    ├─ 测试精确描述 bug 的现象
    ├─ 运行 → 确认测试失败（证明 bug 存在）
    └─ 如果测试通过 → 要么 bug 不存在，要么测试没写对
    │
    ▼
GREEN：应用修复 → 运行测试
    ├─ 复现测试变绿（bug 已修复）
    └─ 其他测试仍然是绿色（没有 regression）
    │
    ▼
同类搜索：Grep 搜索相同的 bug 模式
    ├─ 同样的错误处理模式在其他地方出现吗？
    ├─ 同样的空值忽略在其他方法出现吗？
    └─ 找到 → 一并修复或记录为技术债
    │
    ▼
REFACTOR：清理 → 提交 → 触发 closing-the-loop
```

## 测试反模式

| 反模式 | 为什么是反模式 | 正确做法 |
|--------|--------------|---------|
| 测试依赖外部服务 | 不稳定、慢 | Mock 外部依赖 |
| 测试之间有顺序依赖 | 一个失败导致级联失败 | 每个测试独立 |
| 测试方法名不表达意图 | 看不懂测什么 | `should_throw_when_email_is_empty()` |
| 只测 happy path | 80% 的 bug 在边界 | 覆盖 null、空列表、极大值 |
| 为了覆盖率写测试 | 假的安全感 | 测试行为，不测试实现细节 |
| 在生产代码中写 `if (testing)` | 污染代码 | 用依赖注入 |
| 写完代码再补测试 | 测试变成"验证实现"而非"验证行为" | 先写测试，RED 阶段的设计思考是 TDD 价值的一半 |
| 测了功能没测隔离 | 用户A能看到用户B的数据 | 4 类场景每类至少 2 条测试 |

## Red Flags — 你在跳过 TDD

| 你的想法 | 真相 |
|---------|------|
| "这个太简单了，不需要测试" | 简单代码也会有边界 bug。简单 = 测试更简单 = 没有借口。 |
| "先写实现，测试后补" | 后补的测试往往变成"验证实现"而非"验证行为"。TDD 的价值一半在 RED 阶段的设计思考。不先写测试 = 你没想清楚这个代码到底要干什么。 |
| "我知道这个方法签名" | 看 L4 确认。15% 的概率你记得是错的。花 30 秒验证 > 花 30 分钟 debug。 |
| "这是修 bug，直接改就行了" | 不先写复现测试 = 你不知道 bug 的真正根因。先写测试 → 证明 bug 存在 → 修复 → 证明 bug 消失。跳过测试 = 你不知道是不是"修好了"还是"恰好不报错了"。 |
| "我手动测过了，功能正常" | 手动测试只测了你想到的 1 条路径。TDD 强迫你测 4 类场景 × 2 条 = ≥8 条路径。手动测 ≠ 自动化回归。下次别人改了相关代码，你的手动测试不会自动跑。 |
| "写测试比写代码还花时间" | 写测试的时间 < debug 的时间。第一次不写测试省 10 分钟 → 上线后出 bug → 排查 2 小时 → 修复 → 回归测试 → 总共花了 4 小时。先写测试总共花 20 分钟。 |
| "要搭测试环境太复杂" | 单元测试不需要环境（Mock 外部依赖）。集成测试才需要。如果是单元测试，没有环境借口。如果是集成测试，至少写单元测试覆盖核心逻辑。 |
| "就改一行，不用测" | 一行改错可以毁掉整个服务。至少写 1 个测试覆盖改动的行为。如果改动真的"不值得测"，那改动也不值得做。 |
| "brainstorming 没产出测试场景" | 回到 brainstorming 补全测试场景再进 TDD。没有测试场景 = 你不知道自己在测什么。 |

## 关键行为准则

- **在写实现代码之前，写测试代码**（这是 TDD 的铁律）
- **测试方法名表达意图**：`should_xxx_when_yyy`，不是 `test1`, `test2`
- **如果测试本身写错了**：先修测试，确认测试变绿不是因为"测错了"
- **不确定 API 签名时**：先查 L4 源码，不要凭记忆写
- **每通过一个 RED-GREEN-REFACTOR 循环 → 提交一次**：commit message 标注阶段
- **如果 brainstorming 没产出测试场景 → 退回去补**：TDD 不负责设计测试场景

---

## 完成标准

你必须逐项确认以下所有项：

- [ ] 1. 已从 brainstorming 加载测试场景（4 类场景每类 ≥2 条）
- [ ] 2. L4 签名已验证（至少 3 个关键方法签名与源码一致）
- [ ] 3. ≥8 个测试方法已写出（4 类 × 每类 ≥2 条）
- [ ] 4. RED 阶段：全部测试已运行且至少一个失败（不是测试写错）
- [ ] 5. GREEN 阶段：全部测试通过（包含 RED 阶段失败的那些）
- [ ] 6. L4 签名验证已完成（GREEN 后验证了至少 3 个方法）
- [ ] 7. REFACTOR 阶段已完成（消除了重复、改善了命名）
- [ ] 8. 每次 RED/GREEN/REFACTOR 已分别 git commit
- [ ] 9. Bug 修复场景：同类搜索已完成（Grep 了相同的错误模式）
