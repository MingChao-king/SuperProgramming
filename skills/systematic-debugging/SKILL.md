# Systematic Debugging：四阶段根因分析 + L4 仲裁

## 技能类型：刚性

严格按阶段执行。跳过阶段 = 治疗症状而非根除病因。

---

## 前置步骤：Mini Brainstorming（必做）

**进入四阶段调试之前，先调用 `superprogramming:brainstorming` 做 mini 版**，产出写入 roadmap：
- 澄清 bug 的精确现象和影响范围
- L4 勘探涉及的类/方法（含最近的 git log）
- 测试场景设计（这个 bug 暴露了哪些没测到的场景？）
- 修复大纲（打算怎么修？涉及哪些文件？）

**原因**：不思考直接改 → 治标不治本 → 修一个场景漏同类场景 → 下次会话从零开始。

---

## 四阶段过程

### 阶段 1：复现（Reproduce）

```
→ 找到可靠的重现步骤
→ 确认为什么之前没被发现
→ 写一个失败的测试来捕获它（TDD RED）
```

### 阶段 2：隔离（Isolate）

```
→ 二分法缩小范围：注释掉一半代码 → bug 还在吗？
→ 检查最近变更（git log / git bisect）
→ 确认是代码 bug 还是环境/配置/数据问题
```

### 阶段 3：根因（Root Cause）

```
→ 不是"修好症状"，问 5 次"为什么"
→ ★ L4 为唯一真相源：检查实际方法签名（不是你记忆中的）
→ 检查调用链每层的实际行为
→ L3 踩坑记录是否已有类似问题？
```

### 阶段 4：修复 + 防御（Fix + Defense）

```
→ 应用修复 → 测试通过（TDD GREEN）
→ 同类搜索：Grep 相同的方法调用模式 → 一并修复或记技术债
→ 增加防御层：输入验证 + 错误处理 + 回归测试
→ 回写 L3 踩坑记录（日期 + 问题 + 原因 + 解决 + 严重度）
→ ★ 触发 closing-the-loop：bug 修复也是编码任务，钢印 5 项一个不能少
```

---

## L4 仲裁

调试中对代码行为的任何假设必须对照 L4 验证：

```
你以为：getUser() 返回 null 时会抛异常
L4 实际：getUser() 返回 Optional<User>，不抛异常
→ 假设错了，bug 在别处
```

---

## Red Flags

| 你的想法 | 真相 |
|---------|------|
| "我觉得问题出在 X" | "觉得"不是调试方法。用二分法隔离，用 L4 确认。 |
| "改一下这行试试" | 随机尝试浪费时间。每次改动要有明确假设和验证。 |
| "bug 修好了，不用走 closing-the-loop" | bug 修复 = 编码任务。钢印 5 项一个不能少。 |
| "这是紧急 bug，不用走 brainstorming" | 紧急修复才最需要纪律。mini brainstorming 只需 2 分钟。 |
