# Systematic Debugging：四阶段根因分析 + L4 仲裁

## 技能类型：刚性

严格按阶段执行。跳过一个阶段 = 治疗症状而非根除病因。

---

## 前置步骤：Brainstorming（必做）

**在进入四阶段调试之前，必须先做 mini brainstorming，并把产出写入 roadmap：**

```
bug 报告 / 发现问题
    │
    ▼
mini brainstorming
    ├─ 澄清：bug 的精确现象是什么？影响范围有多大？
    ├─ L4 勘探：涉及哪些类/方法？最近的 git log 谁改过这里？
    ├─ 测试场景：这个 bug 暴露了哪些没测到的场景？（多用户隔离？边界？并发？）
    └─ 修复大纲：打算怎么修？涉及哪些文件？影响哪些功能？
    │
    ▼
将以上 4 项写入 knowledge-base/{项目名}/{项目名}-roadmap.md
（格式：| 日期 | 步骤 | bug描述+根因假设+测试场景+修复文件 | 涉及文件 | 待验证 |）
    │
    ▼
进入四阶段调试
```

**为什么必须先 brainstorming：**
- 不复现直接猜 → 治标不治本
- 不勘探直接改 → 可能引入新 bug
- 不想测试场景 → 修了一个场景，漏了同类场景（这正是用户反馈的问题）
- **不写入 roadmap → 下次会话 agent 看不到，同一类 bug 的分析从零开始**

## 四阶段过程

### 阶段 1：复现（Reproduce）

```
→ 找到可靠的重现步骤
→ 确认为什么这个 bug 之前没有被发现
→ 写一个失败的测试来捕获它
```

### 阶段 2：隔离（Isolate）

```
→ 二分法缩小范围：注释掉一半代码 → bug 还在吗？
→ 检查最近的变更（git log / git bisect）
→ 确认是代码 bug 还是环境/配置/数据问题
```

### 阶段 3：根因（Root Cause）

```
→ 不是"修好症状"，而是问 5 次"为什么"
→ ★ L4 为唯一真相源：
    - 检查相关类的实际方法签名（不是你以为的签名）
    - 检查调用链中每一层的实际行为
    - L3 缓存中的踩坑记录是否已有类似问题？
```

### 阶段 4：修复 + 防御（Fix + Defense）

```
→ 先写测试证明修复有效
→ 应用修复 → 确认测试通过
→ 防御：同样的根因在其他地方会发生吗？
    - 搜索类似代码模式（Grep 相同的方法调用模式）
    - 如果找到了 → 一并修复或记录为技术债
→ 回写 L3 踩坑记录（日期 + 问题 + 原因 + 解决 + 严重度）
→ ★ 触发 `closing-the-loop`：bug 修复也是编码任务，必须走钢印 5 项（代码推送 + 知识库 pull + L3 更新 + roadmap 追加 + 知识库推送）
```

---

## AgentProgramming 增强：L4 仲裁

在调试过程中，任何对代码行为的假设都必须对照 L4 验证：

```
你以为：getUser() 返回 null 时会抛异常
L4 实际：getUser() 返回 Optional<User>，不会抛异常
→ 你的假设错了，bug 在别处
```

---

## 防御深度（Defense in Depth）

修复 bug 后，增加防御层：

1. **输入验证**：在边界处验证输入
2. **错误处理**：给异常路径添加明确的处理逻辑
3. **日志**：在关键路径增加日志（如果之前缺失）
4. **测试**：增加回归测试覆盖这个 bug

## Red Flags

| 你的想法 | 真相 |
|---------|------|
| "我觉得问题出在 X" | "觉得"不是调试方法。用二分法隔离，用 L4 确认。 |
| "改一下这行试试" | 随机尝试是浪费时间。每次改动都要有明确的假设和验证。 |
| "这个问题太小了不值得记" | 记入 L3 踩坑记录。同样的坑会发生在下一个 Agent 身上。 |
| "bug 修好了，不用走 closing-the-loop" | bug 修复 = 编码任务。钢印 5 项一个都不能少：代码 push + 知识库 pull + L3 更新 + roadmap 追加 + 知识库 push。 |
| "修完顺手就结束了，不需要再调技能" | 不调 closing-the-loop = 没有回写。下次遇到同样的坑你还是会踩一遍。 |
| "这是紧急 bug，直接修不用走 brainstorming" | 紧急修复才最需要纪律。mini brainstorming 只需 2 分钟——澄清现象+勘探相关代码+想测试场景。不思考就改 = 可能引入更严重的 bug。 |
