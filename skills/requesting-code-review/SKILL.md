<EXTREMELY-IMPORTANT>
此技能是刚性技能。8 个 reviewer 一个都不能跳过。

TDD 完成不代表代码正确——编译通过不代表没有 bug。本技能是你写代码的"反方律师"：8 个独立视角同时攻击你的代码，通过才能继续。

P0（致命）和 P1（严重）必须清零。P2 应修复但可商议。P3 记录即可。

如果你在想"8 个 reviewer 太多了，3-4 个就够了"——读 Red Flags 第一条。
</EXTREMELY-IMPORTANT>

# Requesting Code Review：8 维度极致审查

## 技能类型：刚性

严格遵循。8 个 reviewer 全部派发，P0/P1 不清零不进入 verification。

---

## 核心哲学

**你的盲区 = 你写代码时的假设。8 个独立 reviewer 各自不带你的假设，从不同角度攻击代码。一个 reviewer 可能漏掉的问题，8 个 reviewer 交叉覆盖，极大概率全部捕获。**

```
                   ┌─ Reviewer 1: 正确性深挖
                   ├─ Reviewer 2: 空安全/边界
                   ├─ Reviewer 3: 安全审计
TDD REFACTOR → 8 ─┼─ Reviewer 4: 并发安全      → 汇总 → 修复 → 再审 → verification
                   ├─ Reviewer 5: 错误处理
                   ├─ Reviewer 6: 性能/资源
                   ├─ Reviewer 7: 回归风险
                   └─ Reviewer 8: 契约验证
```

---

## 触发时机

```
brainstorming → writing-plans → TDD(RED→GREEN→REFACTOR) → ★本技能★ → verification → closing-the-loop
```

**TDD REFACTOR 提交后立即触发。** 不是"合并前"——那是旧版本的位置。审查越早，修复成本越低。

---

## 严重度定义

| 级别 | 含义 | 示例 | 行动 |
|------|------|------|------|
| **P0 致命** | 安全漏洞、数据丢失、必现崩溃 | SQL 注入、无条件 NPE、死锁、用户数据泄露 | **阻塞**，立即修复 |
| **P1 严重** | 逻辑错误、回归风险、边界缺失 | 用户A可访问用户B数据、必填字段无校验、事务边界错误 | **阻塞**，必须修复 |
| **P2 主要** | 模式偏离、重复代码、命名问题、欠佳错误处理 | 未复用已有工具类、方法名不表意、异常吞掉不记日志 | 应修复，可记录 |
| **P3 建议** | 更好的写法、可选优化、风格偏好 | Stream 替代 for-loop、提取常量、加注释 | 可选，记录后续 |

**规则**：P0+P1 必须清零才能进入 verification。P2 可在主代理确认后记录为技术债。

---

## 8 个 Reviewer 并行派发

全部 8 个 reviewer 同时派发。每个使用 5 段式 prompt，只读，独立判断。

### Reviewer 1 — 正确性深挖

```
SCOPE: 本次改动的所有源文件（含新增和修改），逐行审查

GOAL: 追踪每条代码路径，验证逻辑正确性
  - 条件分支是否覆盖所有情况？（if 有 else 吗？switch 有 default 吗？）
  - 循环终止条件是否正确？（会不会死循环？off-by-one？）
  - 类型转换是否安全？（Long→Integer 会溢出吗？）
  - 比较操作是否正确？（== vs equals？引用比较 vs 值比较？）
  - 返回值在所有路径上都是正确的吗？

CONSTRAINTS:
  - 不信任方法名——打开被调用方法的源码验证实际行为
  - 不信任注释——注释说"永远不为 null"不代表真的不为 null

DO NOT:
  - 不要审查命名、风格、注释质量（那是 Reviewer 2/8 的职责）
  - 不要提出"更好的实现方式"建议（那是 Reviewer 8 的职责）
  - 只关注"这段代码逻辑对不对"

RETURN:
  - 问题列表（文件:行号 + 具体问题 + P0/P1/P2/P3）
  - 无问题时返回："正确性：通过"
```

### Reviewer 2 — 空安全/边界条件

```
SCOPE: 本次改动的所有源文件，特别关注方法参数、返回值、集合操作

GOAL: 验证每个可能为 null 的路径和每个边界条件
  - 方法参数为 null 时会怎样？（标注 @Nullable 了吗？有 null 检查吗？）
  - 方法返回值可能为 null 吗？调用者处理了吗？
  - 集合操作：空列表、null 列表、越界访问
  - 数值边界：除零？溢出？负数传入只接受正数的方法？
  - 字符串边界：null、空串、超长串、特殊字符
  - 日期/时间边界：null、过去/未来极值

DO NOT:
  - 不要审查业务逻辑正确性（那是 Reviewer 1 的职责）
  - 如果参数在业务上不可能为 null（例如框架保证），注明并跳过

RETURN:
  - 问题列表（文件:行号 + null/边界场景 + 风险 + P0/P1/P2/P3）
  - 无问题时返回："空安全/边界：通过"
```

### Reviewer 3 — 安全审计

```
SCOPE: 本次改动的所有源文件 + 配置文件

GOAL: 发现安全漏洞
  - SQL 注入：拼接 SQL？MyBatis ${} 而非 #{}？
  - 认证/授权：接口有鉴权注解吗？权限校验遗漏？
  - 数据泄露：敏感字段（密码/手机号/身份证）返回到前端？
  - 日志泄露：日志中打印了密码/token/个人信息？
  - 输入校验：用户输入有 @Valid 吗？有长度/格式限制吗？
  - 文件操作：路径遍历？任意文件读取？

DO NOT:
  - 不要审查非安全问题

RETURN:
  - 漏洞列表（文件:行号 + 漏洞类型 + 攻击场景 + P0/P1）
  - 无问题时返回："安全：通过"
  - 安全漏洞最低 P1；可能造成数据泄露/系统入侵的最低 P0
```

### Reviewer 4 — 并发/线程安全

```
SCOPE: 本次改动的 @Service/@Controller/@Component/@Repository 类

GOAL: 发现并发问题
  - 共享可变状态：Spring 单例 Bean 中有非线程安全的实例变量吗？
  - 竞态条件：check-then-act 操作有原子性保证吗？
  - 事务隔离：并发读写同一数据时隔离级别够吗？
  - 幂等性：重复提交（网络重试、用户双击）会创建重复数据吗？
  - 锁顺序：多个锁的获取顺序一致吗？（会死锁吗？）

DO NOT:
  - 如果类是无状态的（纯函数、无实例变量），注明并跳过
  - 不要审查非并发问题

RETURN:
  - 问题列表（文件:行号 + 并发场景 + 潜在后果 + P0/P1/P2）
  - 无问题时返回："并发安全：通过"
```

### Reviewer 5 — 错误处理完整性

```
SCOPE: 本次改动的所有源文件

GOAL: 验证错误处理是否完整
  - 每个可能抛异常的操作都有 try-catch 或向上声明吗？
  - catch 块是空吞掉异常还是正确处理？（空 catch = P1 最低）
  - 异常类型对吗？（不要 catch(Exception) 吞掉所有）
  - 错误信息对调用者有用吗？（"出错了"不够）
  - 错误信息泄露敏感数据吗？（SQL 错误、堆栈信息返回前端？）
  - 依赖服务不可用时降级了吗？有超时设置吗？

DO NOT:
  - 不要审查异常处理的代码风格

RETURN:
  - 问题列表（文件:行号 + 异常场景 + 当前处理 + 缺失 + P0/P1/P2）
  - 无问题时返回："错误处理：通过"
```

### Reviewer 6 — 性能/资源管理

```
SCOPE: 本次改动的所有源文件

GOAL: 发现性能问题和资源泄漏
  - N+1 查询：循环内调用数据库/远程服务？
  - 资源泄漏：Stream/Connection/IO 未在 finally/try-with-resources 关闭？
  - 不必要的大对象创建：循环内 new 大对象？
  - 全表扫描/无索引查询：新 SQL 或新集合操作有性能隐患？
  - 缓存缺失：频繁计算或查询的结果可以缓存吗？

DO NOT:
  - 不要审查不属于本改动引入的已有性能问题（可注明"另外注意到"）
  - 不要做微观优化建议（如"StringBuilder 比 + 快"）

RETURN:
  - 问题列表（文件:行号 + 性能场景 + 影响 + P1/P2/P3）
  - 无问题时返回："性能/资源：通过"
```

### Reviewer 7 — 回归风险

```
SCOPE: 本次修改的接口/类 + Grep 搜索结果（谁在调用这些接口/类）

GOAL: 评估改动对现有功能的影响
  - 修改的方法签名变了吗？所有调用者更新了吗？
  - 修改的 DTO 字段名/类型变了吗？序列化兼容吗？
  - 修改的数据库表/字段：现有数据兼容吗？需要迁移脚本吗？
  - 修改的配置项：所有环境都有对应值吗？
  - 删除/重命名了公开方法？有遗留调用者吗？

DO NOT:
  - 不要审查新代码的质量——只关注"改了会影响谁"
  - 如果调用者列表为空（新功能），注明"无回归风险"

RETURN:
  - 调用者清单 + 每个调用者的影响评估
  - 问题列表（文件:行号 + 风险 + P1/P2）
  - 无问题时返回："回归风险：通过"
```

### Reviewer 8 — 契约/接口验证

```
SCOPE: 新增/修改的 API 接口 + DTO + L3 experience.md

GOAL: 验证契约一致性
  - API 路由符合项目路由规则吗？（对照 CLAUDE.md § Code Patterns）
  - 返回格式是项目统一的 Result<T> 吗？
  - 方法签名与设计大纲/任务描述一致吗？
  - DTO 字段名与 API 文档/L3 记录一致吗？
  - HTTP 方法对了吗？（GET/POST/PUT/DELETE）
  - 请求参数注解对了吗？（@RequestBody/@RequestParam/@PathVariable）

DO NOT:
  - 不要审查实现质量——只关注"契约是否一致"
  - 没有 L3 记录时不报错，标注"L3 无对应记录，建议更新"

RETURN:
  - 不一致项列表（文件:行号 + 实际 vs 期望）
  - 无问题时返回："契约验证：通过"
```

## 主代理汇总流程

8 份报告全部返回后，主代理执行：

```
Step 1: 去重
  同一问题被多个 reviewer 发现 → 合并为一条（引用所有发现者）

Step 2: 排序
  按严重度：P0 > P1 > P2 > P3
  
Step 3: 决策
  P0+P1 = 0 → 进入 verification
  P0+P1 > 0 → 实施修复 → 再审（Review Loop）
  
Step 4: 报告
  向用户展示：审查摘要 + P0/P1 详情 + P2/P3 概要
```

---

## Review Loop（修复-再审循环）

```
主代理汇总 → P0/P1 问题列表
    │
    ▼
Implementer 逐条修复（只修 P0/P1，不趁机重构）
    │
    ▼
仅重新派发受影响的 reviewer（不是全部 8 个）
  → 修复了 Reviewer 1 发现的问题 → 只重新派发 Reviewer 1
  → 修复了 Reviewer 1 和 3 都发现的问题 → 重新派发 Reviewer 1 和 3
    │
    ├─ 通过 → 进入 verification
    ├─ 仍有 P0/P1 → implementer 再次修复
    │   └─ ≤3 轮
    └─ 第 3 轮仍有 P0/P1 → STOP
        → 主代理介入决策：
          ① 任务描述有歧义？（回头修 task spec）
          ② Reviewer 标准过严？（主代理判断是否降级）
          ③ 问题确实难修？（拆分出独立修复任务）
```

**规则**：
- 每轮只修上一轮 reviewer 指出的 P0/P1
- 修复后 git commit（标注 "fix: review round N"）
- P2 可在主代理确认后记录为技术债，不阻塞
- P3 记录即可，主代理决定是否采纳

## 特殊情况

### 小改动（≤1 文件，≤30 行新增）

仍须派发全部 8 个 reviewer，但每个 reviewer 的审查范围缩小到该文件。小改动也可能有大问题。

### 发现不相关问题

任一 reviewer 发现与本次任务**无关**但严重的问题（P0/P1），必须报告。格式：
`"另外注意到 [文件:行号]：[具体问题]。[建议修复方案]。需要我修吗？"`

**不擅自修改**不属于当前任务范围的代码。

## Red Flags — 你在跳过审查

| 你的想法 | 真相 |
|---------|------|
| "8 个太多，3-4 个够了" | 你跳过的那 4 个维度就是 bug 逃逸的通道。一个 reviewer 看正确性，另一个看空安全——交叉覆盖才有保证。上次你"觉得够了"，这次用户就在抱怨反复修 bug。 |
| "改动太小，快速过一下" | 一行改错花 2 小时排查。小改动也要 8 维度——只是每个 reviewer 审查范围更小。 |
| "我自己审过了，不需要 reviewer" | 你审自己的代码有盲区——你带着实现时的假设在读代码。独立 reviewer 没有这些假设。 |
| "P2 也修了吧，顺便改" | Review loop 期间只修 P0/P1。P2 在 loop 通过后再处理，避免一次改太多引入新问题。 |
| "第 3 轮还不通过，reviewer 太严了" | 可能 reviewer 确实过严，也可能是任务描述有歧义。这正是需要主代理介入判断的时候——不要自作主张跳过。 |
| "某些 reviewer 肯定没发现问题，跳过省时间" | 你不知道哪个 reviewer 会发现问题——这正是审查的意义。全派发。 |
| "这个 reviewer 的问题上次已经修过了" | 重新派发该 reviewer 确认修复——不要在对话里凭记忆判断"修好了"。 |

## 完成标准

你必须逐项确认以下所有项：

- [ ] 8 个 reviewer 全部派发（并行）
- [ ] 8 份报告全部收到（无遗漏）
- [ ] 主代理已完成去重和排序
- [ ] P0 数量 = 0
- [ ] P1 数量 = 0（或主代理明确降级并记录理由）
- [ ] Review loop 通过（≤3 轮）
- [ ] P2 问题已记录（修复或技术债）
- [ ] P3 建议已处理（采纳或放弃）
- [ ] 审查摘要已向用户展示
