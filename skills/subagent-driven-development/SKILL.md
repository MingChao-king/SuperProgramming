# Subagent-Driven Development：并行子代理执行

## 技能类型：刚性

≥2 个独立任务时，必须使用子代理并行。不能以"任务不多"为借口串行。

---

## 何时必须使用

以下条件**全部满足**时，必须派发子代理：

1. 任务清单中有 ≥2 个任务
2. 这些任务**互不依赖**（任务 A 的输出不是任务 B 的输入）
3. 每个任务在 writing-plans 中已有明确的文件路径和验证方式

---

## 执行模式

```
任务清单（来自 writing-plans）
    │
    ▼
按依赖关系分组 → 互不依赖的并行组同时派发
    │ → 子代理接收：任务描述 + 文件路径 + 验证方式 + 复用组件
    ▼
两级审查
    ├─ 第一级：规格合规（产出是否符合任务描述？文件路径对吗？验证通过吗？）
    └─ 第二级：代码质量（是否符合项目现有模式？有无重复造轮子？）
```

### 派发示例

```
Agent(
  description: "创建 UserProfileDTO",
  prompt: "
    任务：创建 UserProfileDTO
    文件：src/main/java/.../dto/UserProfileDTO.java
    要点：字段 nickname, avatar, email，@Data + @Builder
    验证：编译通过 + JSON 序列化含三个字段

    约束（来自项目 CLAUDE.md）：
    1. 写之前搜索项目中是否已有类似 DTO
    2. 不确定 API/注解行为时查源码
    3. 只写任务要求的最少代码（YAGNI）
    4. 发现计划外问题立即回报
  "
)
```

### 审查不通过怎么办

| 级别 | 不通过处理 |
|------|-----------|
| 规格合规不通过 | 修正任务描述，重新派发 |
| 代码质量不通过 | 主代理修正或指导子代理修正 |

---

## 关键行为准则

- **子代理失败** → 分析原因，调整任务描述，重新派发
- **产出偏离计划** → 对比计划和产出，决定更新计划还是修正产出
- **不要强行拆分有依赖的任务** → 依赖任务必须串行
- **派发后主代理不要闲着** → 处理有依赖的任务或准备下一阶段

---

## Red Flags — 你在逃避并行

| 你的想法 | 真相 |
|---------|------|
| "我一个个串行做也很快" | 2 个独立任务串行 = 2x 时间。并行 = 1x。 |
| "就两个任务，不需要子代理" | 2 个独立任务就值得派发。门槛是 2，不是 5。 |
| "子代理也需要上下文，不如我自己做" | 任务描述已含所有上下文（文件路径 + 验证 + 复用组件）。 |
| "子代理质量不可控" | 两级审查机制就是为此设计的。 |

---

## 完成标准

- [ ] 任务清单已按依赖关系分组（识别并行组）
- [ ] 每个并行组已派发子代理
- [ ] 每个子代理产出已通过两级审查
- [ ] 子代理发现的计划外问题已处理
- [ ] 所有并行任务已完成，产出已整合
