# Closing the Loop：回写闭环

## 技能类型：刚性

严格遵循。未完成闭环 = 任务未交付。

---

## 核心哲学

**每一次编码不只是完成当前任务，而是为未来所有项目积累资产。**

```
没有闭环：每次任务 → 冷启动 → 从零开始
有闭环：  每次任务 → 命中已有模式 → 快速完成 → 回写新经验
```

---

## 两个仓库 — 什么东西往哪放

| 仓库 | 远程 | 存放内容 |
|------|------|---------|
| **代码仓库** | 项目自身的 `git remote` | 源码、配置、CLAUDE.md |
| **知识库** | `https://gitee.com/gudu-code-man/agent-programming-mode.git` | L1/L2/L3/Roadmap |

知识库目录结构见项目 CLAUDE.md § Knowledge Base。

### ⚠ 常见错误

| 错误 | 正确做法 |
|------|---------|
| 把 roadmap.md 推到代码仓库 | 放在 `knowledge-base/{项目名}/` |
| 把 CLAUDE.md 推到知识库 | CLAUDE.md 是项目配置，放在代码仓库 |
| 只在对话里写大纲，不保存文件 | 保存到 `knowledge-base/{项目名}/{项目名}-roadmap.md` |

## 子代理委派（推荐）

L2/L3/Roadmap 文件可能数万行，全部加载浪费上下文。**推荐将 L3/Roadmap/L2 回写委派给子代理**：

```
主代理：准备任务摘要 + 推送代码仓库
    │
    ▼
子代理：KB pull → 读取 L3 + roadmap + L2 → 判断更新 → 执行编辑 → KB add + commit + push
```

子代理 prompt：
```
你是 closing-the-loop 回写子代理。请完成知识库回写：

任务摘要：
- 做了什么：[描述]
- 涉及文件：[列表]
- 新增/修改的关键类/API/配置：[列表，如无写"无"]
- 踩到的坑：[描述，如无写"无"]
- 跨项目经验：[描述，如无写"无"]

你的工作：
1. cd knowledge-base && git pull --ff-only（拉取最新，避免冲突）
2. 读取 knowledge-base/{项目名}/{项目名}-experience.md，判断更新，执行编辑
3. 读取 knowledge-base/{项目名}/{项目名}-roadmap.md，追加一行实现记录
4. 检查 knowledge-base/通用编码经验.md（按 CLAUDE.md § Write-Back Criteria），执行编辑
5. cd knowledge-base && git add -A && git commit -m "roadmap: {项目名} - <描述>" && git push
6. 回报：修改了哪些文件 + git push 结果
```

**主代理只需推送代码仓库**（第 1 项）。知识库的 pull、编辑、commit、push 全部由子代理完成。主代理确认子代理回报"推送成功"即完成闭环。

---

## 钢印 5 项（主代理只做第 1 项，第 2-5 项由子代理完成）

### 1. 项目代码推送（主代理执行）

```bash
cd {项目目录} && git add -A && git commit -m "{描述}" && git push
```
判据：`git status` 干净。

### 2. 知识库拉取（子代理执行）

子代理 prompt 中已包含 `git pull --ff-only`。主代理不需操作。

判据：子代理回报"已拉取最新"。

### 3. L3 项目速查更新

判断是否更新：
```
新增了关键类/API/配置/模式？踩到了坑？→ 任一 → 更新 L3
```

**L3 必须包含 6 个强制模块，≥150 行**：

| 模块 | 内容 |
|------|------|
| 项目结构速查 | 所有模块/服务 + 端口 + 包名 + 职责 |
| 关键类速查 | ≥10 个核心类，含完整路径和说明 |
| API 契约速查 | 路由规则、鉴权白名单、Feign 接口签名 |
| 代码模式 | 3-5 个最常用代码写法 |
| 配置速查 | 各服务关键配置项 |
| 依赖速查 | 关键依赖 artifactId 和用途 |
| 踩坑记录 | 每个坑标注严重度(S/M/L)、日期、是否已修复 |

L3 是 L4 的缓存。验证标记：✅(已验证) ⚠️(超期) ❌(签名已变)。完整同步规则见 CLAUDE.md § Cache Sync。

### 4. Roadmap 追加

每次任务追加一行：
```markdown
| 日期 | 步骤 | 做了什么 | 涉及文件 | 验证结果 |
|------|------|---------|---------|---------|
| 06-21 | 5 | 新增 UserProfileDTO | dto/UserProfileDTO.java, controller/UserController.java | ✅ 测试通过 |
```

### 5. 知识库推送（子代理执行）

子代理 prompt 中已包含 `git add -A && git commit && git push`。主代理不需操作。

判据：子代理回报"推送成功"。

## L2 回写判据（必查）

**本次任务有没有值得其他项目借鉴的经验？**

```
发现依赖版本陷阱？框架配置反模式？跨项目通用解法？技术坑？文档误导？性能优化？
  ├─ 任一 → 回写 L2（通用编码经验.md）
  └─ 一个都不沾 → 明确说"本次无 L2 回写"
```

L2 格式：✅ 推荐 + ❌ 避免 + 代码对比。参考字段用技术概念（如 "jjwt 0.12.x"），不用项目名。
完整判据见 CLAUDE.md § Write-Back Criteria。

## Closing the Loop（mini）

配置变更（改 yml/properties/环境变量）不产生新类/API/模式，跳过 L3 和 L2：

| 钢印项 | 完整版 | mini 版 |
|--------|--------|---------|
| 代码 push | ✅ | ✅ |
| 知识库 pull | ✅ | ✅ |
| L3 更新 | ✅ | ❌ |
| Roadmap 追加 | ✅ | ✅ |
| 知识库 push | ✅ | ✅ |
| L2 检查 | ✅ | ❌ |

配置涉及新技术知识（如版本差异）→ 升级为完整版。

## Red Flags — 你在跳过闭环

| 你的想法 | 真相 |
|---------|------|
| "这次改动太小了，不值得记" | 最宝贵的经验来自小改动。不记 = 下次还会踩。 |
| "太累了，下次再写 L3" | L3 不写 = 下次冷启动。现在的"累"换不来下次的"快"。 |
| "KB push 我手动做就行" | 手动做 = 会忘。子代理一条龙（pull → 编辑 → push），零遗漏。 |
| "回写我自己做就行，不用子代理" | L2/L3/Roadmap 可能上百万 token。委派 = 省上下文给编码任务。 |
| "先让用户确认完成，回写是额外工作" | 回写是任务的一部分。用户说"完成" = 闭环也完成了。 |

---

## 完成标准

- [ ] 1. 项目代码已 git commit && git push（主代理执行，代码仓库）
- [ ] 2. KB 回写子代理已派发并完成（含 pull → 编辑 L3/roadmap/L2 → commit + push）
- [ ] 3. 子代理回报确认：所有 KB 文件更新 + git push 成功
